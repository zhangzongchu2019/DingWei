# 设计：会话间消息统一投递

- **版本**：**v2（架构重定向）** —— v1 已作废，勿参考
- **状态**：技术设计稿

---

## 1. 架构原则

> **① hub 只是管道，只负责投递。是否需要重新投递，由发送方自己决定。**
> **② sessionHelper 收到，hub 就算投递成功。**
> **③ hub 必须明确告诉发送方：这是网络问题，还是 helper 问题。**
> **④ 除非有特别理由，所有消息都必须进入同一个队列。**

### 1.1 单一队列（原则 ④）

| 输入源 | 进队列 | 说明 |
|---|---|---|
| 飞书消息 | ✅ | 已在走 |
| **会话间消息** | ✅ **本次纳入** | 现在跑在内存 deque 上 |
| **hub 推送的网络指南事件**（`agent_network_skill`） | ✅ **本次纳入** | 它现在**硬插队**（`recv_loop` 里排在 `buffer_if_busy` 之前），唯一理由是「**只有注入成功才能 ack**」。**新语义下 helper 收到即 ack、hub 立刻收手不再重推 ⇒ 插队理由消失。** |
| **网页终端 `/view` 输入** | ❌ **唯一的「特别理由」** | **它是交互字节流，不是消息。** ① 需要 <50ms 回显，过队列 + SQLite 往返毁掉交互；② **CLI 卡死时要靠 `Ctrl-C` 救它——若排队，队列会回「CLI 忙，已排队」，把唯一的救命手也锁死。人的抢占权必须能绕开队列。** |
| provision（装制品） | ❌ | **根本不注入 CLI**，不是「给 CLI 的消息」 |

**⇒ 终局：一个队列（SQLite inbox）、一条循环（`delivery_worker_loop`）、一个注入点。**
**内存 deque（`pending_inbound`）、`pending_drain_loop`、`buffer_if_busy` 全部删除。**

### 1.2 为什么现在是两个队列（历史，非设计）

- **早期 busy buffer 实现**：加入内存 deque。动机见代码注释——*"Route all CLI inbound messages through the drain loop so recv_loop **never blocks** on subprocess/pexpect injection (**#4 recv_loop hang fix**)"*。**它是止血补丁，不是队列设计。**
- **可靠 delivery inbox 实现**：加入 SQLite inbox，为飞书可靠投递。该提交 **77 增 / 2 删**——**纯新增，没有合并旧的**。
- **当时合并不了**：hub 只给飞书盖 `delivery` 标，会话间拿不到 ⇒ 只能继续走内存 deque。
- **⇒ 本次改 hub（会话间也盖 delivery 标），正是让合并成为可能的那一步。**

### 1.3 本次不实现、但**必须预留**的能力

> **本次不做功能，只把字段/状态/协议位留出来 —— 否则将来加就得做 schema 迁移。本项目今日已被 schema 迁移咬过一次（F1/F5）。**

| 能力 | 现状 | 本次预留 |
|---|---|---|
| **出队优先级** | helper inbox 出队是 `ORDER BY received_at, delivery_id` —— **纯 FIFO，无 `priority` 列** | ① 表加 `priority INTEGER NOT NULL DEFAULT 0`<br>② 出队改 `ORDER BY priority DESC, received_at, delivery_id`（**本次 priority 全为 0 ⇒ 行为与现在的 FIFO 完全一致**）<br>③ 信封 `meta.priority` 透传到该列 |
| **撤回 / 取消** | ⚠️ **hub 与 helper 不对称**：hub 的 `session_delivery` **已有** `cancelled` 状态；**helper 的 inbox 没有**——`CHECK(state IN ('received','processing','processed','failed'))` | ① helper inbox 的 state CHECK **加入 `'cancelled'`**<br>② 预留控制信封类型 `meta.type = "delivery_cancel"` + `delivery_id`<br>③ 语义（本次不实现，写进文档）：**只有尚未注入的消息可撤回**；已注入的不可撤回 |

**⇒ 完成该结构后，加优先级 = 改一行 `ORDER BY` + 填值；加撤回 = 实现一个控制信封的处理分支。两者都不再需要 schema 迁移。**

**⚠️ 两个预留能力的边界条件**：

- **撤回 × `processing` 状态**：「只有尚未注入的可撤回」需区分 `processing`（已 claim、注入中）与 `processed`（已注入）。
  `processing` 的撤回本身安全（`release()` 回 `received` 再 cancel），但若 **`delivery_worker` 在注入过程中崩溃 → 消息卡 `processing`** → 需 `_initialize()` 的 reset（`processing → received`）先恢复，才能被 cancel。
  **⇒ 若 cancel 信封到达时消息仍在 `processing`：应 reject 还是入队等 reset？——将来实现时必须明确，本次不实现但记在此。**
- **优先级 × 严格 FIFO**：`ORDER BY priority DESC, received_at` 一旦有人填非零 priority，**高优先级会插队，严格 FIFO 不再成立**。
  **这是预留能力的预期行为，不是 bug** —— 但判据 #8（串行）必须注明「`priority` 全 0」的前提。

**推论（这是本设计的全部）**：

| 层 | 责任 |
|---|---|
| **hub** | 送到 helper 这**一跳**。成功/失败**如实回报发送方**。**零重试、零暂存、零兜底。** |
| **helper** | **收到即负责到底**：落 SQLite、忙时排队、串行注入、重启后从自己的库恢复。 |
| **发送方** | 拿到明确回执，**自己决定要不要重发**。 |

**⇒ 持久化的责任在 helper，不在 hub。** 既然 helper 一收下就算成功，消息的家就在 helper 的 SQLite 里；hub 没有理由再存一份，更没有理由替它重投。

## 2. 问题

会话间消息（**全团队协作的主通道**）跑在 helper 的**内存 deque**（`maxlen=100`）上：不持久化、满了静默丢最老、helper 重启全丢。而**发件人拿到的 `sent` 只代表 hub 收下了**——不代表对方 helper 收下了，更不代表注入了 CLI。

**实测**：给一个**根本不存在**的会话 `example-missing-0001` 发消息，`send.py` 打印 `sent to=… bytes=197`，**一副完全成功的样子**。hub 其实回了「投递失败」错误回执，但 `send.py` **发完即断**（临时会话名 `<short>note`），回执发给一个已断开的连接，**掉进虚空**。

## 3. 关键事实（已核实，勿重复论证）

- **helper 侧无需改动即可承接可靠投递**：所有 helper 均已声明 `delivery_ack_v1`（`app.py:267`）；`is_delivery()` / `accept_delivery()` / SQLite inbox（**按会话隔离**）/ `delivery_worker_loop`（串行 claim）/ busy ack / 按 `delivery_id` 幂等 —— **全部现成**。
- **唯一闸门**：`hub.go:2374 / 2395` 的 `ackInbound = from.Kind == addressFeishu` —— 会话间永远 `false`。
- **hub 已有能力门控与回落**：`if ackInbound && h.sessionSupportsDeliveryACK(c) && …` **else** `c.write(ctx, env)` ⇒ 未声明能力的老 helper 自动回落，不会被打挂。
- **hub 已有错误回执机制**：`hub.go:921` `if err := h.RouteEnvelope(…); err != nil { c.write(errorEnvelope(…, "投递失败："+err.Error())) }` —— **机制在，只是不分类，且发送方收不到**。
- **hub 手里有区分「寻址失败」与「离线」的数据**：`session_endpoint` 表。实测 `example-missing-0001` 在该表 **0 条** ⇒ 可判定为寻址失败。但**现在两种情况说的是同一句话**：`fmt.Errorf("session %s offline", …)`（`hub.go:2927`）。

## 4. 改动

### 4.1 hub：会话间纳入可靠投递（2 行）
`ackInbound` 对会话间也成立（仍受 `sessionSupportsDeliveryACK` 门控）。

### 4.2 hub：**删除全部应用层重试与暂存**

| 删除 | 原因 |
|---|---|
| 指数退避重试（1s→2s→5s→10s→30s→1m→2m） | **网络层的责任，不是 hub 的** |
| `attempts` + **12 次上限** | 不重试了，没有次数 |
| **30 分钟过期判 `dead`**（`sqlite.go:1345-1347`） | 没有滞留消息可过期 |
| `lease_until` + lease 超时 | 没有 in-flight 状态 |
| `drainSessionDeliveries` / `scheduleSessionDeliveryDrain` / AfterFunc 链 | 不暂存了，无可 drain |
| busy → +15s 重投（`hub.go:2989`） | **helper 收下了就是成功** |

**`session_delivery` 表保留，但仅作审计**（记录每条消息的最终状态），不再是重试队列。

### 4.3 hub：**失败分类 + 如实回执**（本设计的核心）

| 面 | 情况 | 判定 | hub 行为 | 回执给发送方 |
|---|---|---|---|---|
| **寻址** | 寻址失败 | `session_endpoint` **无此会话** | 立即失败 | **`寻址失败：不存在会话 X`**（附「请查在线清单」） |
| **网络** | 对方不在线 | `session_endpoint` **有**记录，但不在 `sessionClients` | **不暂存**，立即失败 | **`对方不在线，未投递`** |
| **网络** | 网络失效 | `c.write` 失败 | **不重试**，立即失败 | **`网络投递失败：<原因>`** |
| **helper** | 已注入 | ack `injected` | 收手 | `已注入对方 CLI` |
| **helper** | 对方忙 | ack `busy` | 收手 | `对方忙，已持久化入队，空闲后执行` |
| **helper** | 队列满 | ack `failed(queue_full)` | **收手，不重试** | **`对方队列满（100），未收下 —— 是否重发你自己定`** |
| **helper** | inbox 不可用 | ack `failed(inbox_unavailable)` | **收手，不重试** | **`对方 inbox 不可用（磁盘满等）`** |

### 4.4 helper：`send.py` 等回执再断开
现状：`sock.close()` 后直接 `print("sent")`，**不等任何回执**。
改为：**等 hub 的投递回执**（在新语义下 helper 是**收到即 ack**，等待是毫秒级，不是等 CLI 干完活），并把结果打印出来。

### 4.5 helper：**删除内存 deque 路径**（不是加互斥）

> **⚠️ 订正**：本节旧文写「加互斥锁」，与 §1.1「删除内存 deque」**自相矛盾**——早期修订只改了 §1.1、忘了回头改本节。**以本节现文为准。**

**问题**：`delivery_worker_loop`（读 SQLite）与 `pending_drain_loop`（读内存 deque）**并行运行、各自独立查 `is_idle` 再注入、无互斥** ⇒ CLI 从忙变闲的瞬间两边可能同时注入（**T9 那个 TOCTOU 的同款结构**）。

**修法：删掉，不是加锁。**
- 删 `pending_inbound`（内存 deque）
- 删 `pending_drain_loop`
- 删 `buffer_if_busy`
  （它的止血使命——「不阻塞 `recv_loop`」——由 SQLite 队列接管：`accept_delivery` **落盘即返回**，同样不阻塞。）

**⇒ 只剩一条注入路径，没什么可竞的。** 加锁是给两条并存的路径打补丁；删掉才是回到本来就该有的样子（见 §1.2：两个队列是历史，不是设计）。

**回落路径的处置**：内存 deque 兼着「未声明 `delivery_ack_v1` 的老 helper」的回落职责。当前**所有 helper 均已声明该能力**；**版本强制（`WP_VERSION_ENFORCE=enforce`）切上去后，不合规的老 helper 根本连不上** ⇒ 那时删除彻底安全。
**应删除**（现网无此类 helper）；若需过渡期，可先保留但**必须加互斥**，并标记为待删。

### 4.6 hub：`SourceType` 区分来源
`EnqueueSessionDelivery` 里硬编码 `"feishu_inbound"` ⇒ 会话间也写成 feishu，审计来源全错。

### 4.7 `send.py` 传输方式 = **方案 C（本地 outbox）**

> `send.py` 的传输方式按技术约束确定。

| 方案 | 结论 |
|---|---|
| **A. 临时连接 + 同步等回执** | ❌ **排除**。重新引入回执超时敏感参数（短→假失败→重发→**重复**；长→发送方卡住），且**仍是每次新 id ⇒ 回执丢失照样重复**。 |
| **B. 控制块 `DW_ROUTE_V1` 走长连接** | ❌ **排除**。无超时，但**依赖 helper 解析 CLI 输出** —— 已踩过：claude 的 markdown 缩进让 `^#` 前缀行匹配不上，**消息根本没发出**。那是**静默失败**。**用一个静默失败（send.py 假成功）换另一个静默失败（控制块没被解析），得不偿失。** |
| **C. 本地 outbox** | ✅ **选定**。`send.py` **写本地队列即返回**；helper 的**长连接**取走发出、收 ack、**注入 owner 的 CLI**。<br>不等、不超时、**不解析 CLI 输出**；`from` = **真实会话名** ⇒ hub 能把 ack 路由回来；顺带让 `<short>note` 临时会话名退役（**正是待办 T7**）。 |

### 4.8 🔴 P1【硬依赖，不做则 §4.7 白做】稳定 msg-id + hub 按它去重

**问题（早期论证不成立）**：

早期论证曾称「重复风险 = 回执丢失风险，判据 #10 那道闸门堵住」——**这是错的**。核代码：

```
send.py:81   "id": os.urandom(8).hex()          ← 每次全新
hub.go:2933  deliveryID := "dlv_" + randomHex(16) ← hub 每次路由再生成一个新的
inbox.py:52  PRIMARY KEY(session_name, delivery_id) ← helper 靠它去重
```

⇒ **owner 重发（重跑 send.py）= 全新 `env.ID` = 全新 `delivery_id` = helper 认不出是同一条 = 重复注入。**
⇒ **判据 #9（幂等）只挡「hub 重发同一 `delivery_id`」，不挡「owner 重发」。判据 #10 只是「打印真实结果」，不是去重闸门。**
⇒ **owner 一重发就重复，与回执丢没丢无关。**

**修法（必做）**：
1. **outbox 给每条逻辑消息分配稳定的 `client_msg_id`**（持久化；**owner 重发同一逻辑消息时复用同一个 id**）。
2. **端到端透传**：`send.py` → outbox → helper → hub → helper(收) 。
3. **hub 按 `client_msg_id` 去重**：用它派生 `delivery_id`，或按 `source_message_id` 去重（**改 `INSERT OR IGNORE` 的键 / 加唯一约束**）。

**⇒ 这是一个本设计原先没有的、必做的 hub 改动。**

### 4.9 P2：异步回执格式（方案 C 的必然要求）

C 下 `send.py` **写完 outbox 即返回**，真实结果（寻址失败 / 对方不在线 / 已注入 / 队列满）是**稍后**经长连接注入 owner 的 CLI。

⇒ **回执必须回带 `client_msg_id` + 收件人 + 结论**，否则 owner（尤其飞书里的人）**不知道是哪一条成功/失败**。

**回执格式（待细化）**：
```
【投递回执】msg=<client_msg_id> → <收件全名>：<结论>
结论 ∈ { 已注入 | 对方忙已入队(第N位/深M) | 对方队列满未收下 | 寻址失败 | 对方不在线 | 网络投递失败 | hub 无响应 }
```

### 4.10 P3：outbox 的两条硬约束（否则破坏原则④）

**outbox 不违反原则④** —— 原则④管的是**收件侧**「单一注入队列」（`delivery_inbox` → 单注入点）；**outbox 是发件侧出站，两者正交**。

**但必须守住两条**：
1. **outbox 与 inbox 同等持久化标准**：SQLite/WAL；**重启时把在途 `processing` 恢复为可重发态**（照抄 `inbox._initialize()` 的恢复逻辑）。
2. **🔴 严禁 outbox 的取走逻辑变成第二条 inbound 注入路径** —— **它只出站，不注入本地 CLI**。（否则又造出第二个注入点，原则④当场破功。）

**P4**：outbox **落盘失败**（磁盘满）时，`send.py` 必须**同步**报给 owner（`outbox 写入失败`），**不得静默**。
helper 宕机时 outbox 攒着等它起来——**可接受**（此时 owner 的 CLI 也随之不在）。

## 5. 🔴 待拍板：飞书路径是否同样适用？

**本设计的原则是「hub 只是管道」。若统一适用，飞书 → 会话的行为也会变**：
- 现状：会话离线 → hub **暂存 + 重投**，30 分钟后判 dead（**静默**）
- 新语义：会话离线 → **立即回执「对方不在线，未投递」**，老板自己决定重发

**设计倾向统一适用**：一致性值钱；且现状（暂存 30 分钟然后静默判死）本身就是坏的——**老板根本不知道消息死了**。新语义下他立刻知道。

**但这是对一条在产的路径做行为变更，请拍板。**

## 6. 「发送方」是谁 —— 关键澄清

> **hub 是投递管道，尽最大可能投递，所有失败原因由发送方自己处理。**
> **发送方的最终 owner 是人（飞书）或 CLI（Agent），都有足够的智能判断是否需要进一步投递。**

**⇒ `send.py` 不是发送方，它只是工具。真正的发送方是它背后的 owner。**

早期分析中**把 `send.py` 当成了发送方**，由此得出「发送方没有重试能力」的错误结论。澄清如下：

| 问题 | 在正确理解下 |
|---|---|
| 「`send.py` 是一次性脚本，退出后无人持有消息去重发」 | **不需要它持有。**它只要把回执**呈现给 owner**（打印到 CLI / 回到飞书），**owner 自己决定是否重发**。 |
| 「人肉重发 = 全新 `id` = helper 去不了重 = 重复注入」 | **只有在「消息其实送到了、但回执丢了」时才会重复。** Agent 拿到 `已注入` 就不会重发。<br>**⇒ 重复的风险 = 回执丢失的风险 —— 而这正是判据 #10 那道闸门要堵的。** |
| 「失去异步留言能力」 | **没失去，只是责任转移。** Agent 拿到 `对方不在线` → **可查在线清单、等对方上线再发**。这正是「足够的智能」。 |

**⇒ 因此 §5 一并拍板：飞书路径同样适用纯管道。** 飞书那端的 owner 是人，人看到「对方不在线」同样有智能判断。**唯一要除掉的是「静默判死」——而纯管道 + 如实回执正好除掉它。**

### 6.1 但这带来一个必须补的洞：中心可观测性

**发送方已退出、hub 不暂存 ⇒ 没有任何一方留下「曾经想发这条消息」的痕迹。**
将来「队友说没收到你消息」——**无处可查**。（原设计缺口）

**补法（不违反零暂存）**：
`session_delivery` 表作为**审计**，**记录每一次投递尝试及其结果**（含 `寻址失败` / `对方不在线` / `网络失效` / 各类 ack）。
**审计 ≠ 队列** —— hub **不据此重投**，只是留痕。**⇒ 事后可查「这条消息 hub 投过、结果是什么」。**

### 6.2 仍然存在、需写进文档的代价

1. **重连窗口从「自愈」变「硬失败」**：成员每次重连都有一段「已注册但不在 `sessionClients`」的窗口。旧模型靠暂存自愈；新模型直接失败 ⇒ **正常运维抖动会变成偶发投递失败**，由 owner 重发兜住。
2. **回执超时成为新的敏感参数**：`send.py` 等回执 ⇒ 发送延迟耦合到对端与 hub 的健康（实测 hub 在 DB 争用下 WS 升级可达 **3.75s**）。**超时太短 → 假失败 → 重发 → 重复；太长 → 发送方卡住。** ⇒ **必须显式定义超时值与超时语义**（判据 10b）。
3. **成功 = 送达 helper ≠ 执行**（原则②的必然）：helper 收到即 ack 成功；若之后 CLI 注入永久失败（CLI 死 / 消息畸形），**发送方永不知道它没被执行**。**接受，但写进文档。**
4. **单一队列的队头阻塞**：一条慢注入阻塞其后所有消息（含控制/紧急类）；网络指南事件从「硬插队」降为 FIFO。**队列内无抢占，直到 `priority` 真正启用**（§1.3 已预留）。`/view` 绕开只覆盖交互字节流，**不覆盖「紧急控制消息」**。

## 7. 验收判据

> **⚠️ 判据 #10 是整个设计的闸门 —— 它不通，后面全白做。**
> hub 的分类回执做得再漂亮，若 `send.py` 收不到，**v2 的全部价值归零**。

**hub 行为**
1. **寻址失败**：发给不存在的会话 → 发送方收到 **`寻址失败`**，**不是** `offline`；hub **不暂存、不重试**。
2. **对方不在线**：发给已注册但断线的会话 → 发送方收到 **`对方不在线，未投递`**；hub **不暂存**。
3. **零重试（必须端到端验证，不接受查表）**：
   ⚠️ **查 `session_delivery` 表是静态检查——重试是内存行为，不一定落表，有 bug 也看不出来**。
   **正解**：发消息 → **立即 `iptables` 丢包切断 hub↔helper** → 等 30s → 恢复 → **确认 helper 只收到 1 条**（而非多次重投）。
4. **收到 ack 即终态**：helper 回任意 ack 后，hub **不再发送同一 `delivery_id`**。

**helper 行为**
5. **持久化**：CLI 忙时，消息落进 `~/.dingwei/sessionhelper_inbox-<会话>.db`。
6. **重启不丢**：消息入队后 **kill 并重启 helper**，消息**仍被注入**（helper 自己从 SQLite 恢复；`_initialize()` 会 `UPDATE … state='received' WHERE state='processing'`）。
7. **长忙不丢（改为正向验证）**：CLI 持续忙 **35 分钟** → 空闲 → 消息**成功注入**。
   ⚠️ 不只验「没消失」，**还要验「最终到达」**（旧判据只验证了过期逻辑不存在，没验证消息真的送达）。
8. **串行**（对应 4.5）：**混入不同来源**的消息（飞书 3 条 + 会话间 3 条 + 网络指南事件 1 条，交替到达）→ **注入顺序严格按 `received_at`**。
   ⚠️ 旧判据说「横跨两条注入路径」——**内存 deque 已删，只剩一条路径，无「横跨」可言**（§1.1/§4.5 矛盾）。
   ⚠️ **前提：`priority` 全为 0**（非零时高优先级会插队，届时严格 FIFO 不再成立——这是预留能力的预期行为，不是 bug）。
9. **幂等**：同一 `delivery_id` 重复到达，**不重复注入**。
9b. **busy ACK 收敛**：helper 对同一 `delivery_id` **最多回一次 busy ACK**；**重启后允许重发一次**（已确认可接受：发送方只是多收一条回执，不丢消息）。

**回执可见性（本设计的核心价值 —— 🔴 闸门）**
10. **`send.py` 等到回执再断开**，并打印真实结果，不再无条件 `sent`：
    - `已注入` / `对方忙已入队（第 N 位，队列深 M）` / `对方队列满未收下` / `寻址失败` / `对方不在线`
    - **10a【子项，显式验证】**：`send.py` **确实等到了回执**（不是发完即断）。**这是闸门 —— 不通则 #1~#4 全部无效。**
    - **10b【超时】**（原设计缺失）：hub 无响应超过 **N 秒**（建议 10s）→ 打印 **`hub 无响应，消息未确认投递`**，**不得**打印 `sent`。
11. **队列满不静默**：把对方队列灌到 100，第 101 条 → 发送方**明确收到 `队列满，未收下`**（**不得** DEAD 后无人知晓）。
12. **DEAD 必有回执**：任何走到最终失败的消息，发送方**必须**收到失败回执。

**不回归**
13. **老 helper 回落**：未声明 `delivery_ack_v1` 的 helper 仍能正常收发。
14. **飞书路径**：按 §5 的拍板结果验证（不变 / 或同样立即回执）。

**🔴 去重（对应 §4.8 P1 —— 现状必失败，直到 P1 落地）**
16. **owner 重发同一逻辑消息（同 `client_msg_id`）→ 不得重复注入。**
    ⚠️ **现状必失败**：`send.py` 每次 `os.urandom(8).hex()` 生成全新 `env.ID` → hub 生成全新 `delivery_id` → helper 按 `delivery_id` 去重，认不出是同一条 → **重复注入**。
    ⚠️ **判据 #9（幂等）不覆盖此项** —— 它只挡「hub 重发同一 `delivery_id`」。

**证明「消息真的没丢」**
15. 发件方发 **100 条带单调递增 seq**，中途随机 kill helper 3 次 + CLI 持续忙 5 分钟 → **最终收到的 seq 集合无缺口**。
    > *「日志只告诉你『某时刻发生了什么』，不告诉你『什么没发生』。」*
    ⚠️ **前提：判据 #1~#4 的 hub 回执分类全部正确。** 若分类有 bug——例如把「对方不在线」误报为「已注入」——**seq 对账会给出假阴性**：收方 seq 无缺口，但消息其实没投到。

### 7.1 为什么 v2 比 v1 更容易证明「没丢」

- **v1**：消息可能在「hub 暂存 → 重投 → busy → +15s → 重投 → … → 30min 到 → DEAD」这条**长链**上任一环节静默消失，而发送方只看到最初的 `sent`。证明没丢，需要覆盖 hub 侧状态机的**全部转移**。
- **v2**：链路**极短**——`hub 投递 → helper accept → SQLite → delivery_worker → 注入 CLI`。每个环节都是**二值成败**：
  - hub 投递失败 → 发送方**立即知道**（#1/#2）
  - helper accept 成功 → 消息在 SQLite，**重启不丢**（#6）
  - 注入失败 → 消息仍在 SQLite（`state='received'`），下次 claim 再试
  - CLI 忙 → `release()` 回 `received`，循环等待
- **唯一的新风险 = hub ↔ `send.py` 的回执链路。** ⇒ **判据 #10 是闸门。**
