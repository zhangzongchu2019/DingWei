# DingWei 部署与冒烟说明

本文面向裸机 Linux + systemd + nginx 部署。Docker 仍可用于本地构建验证，但生产推荐用本流程，避免手工拼装遗漏 `WP_SECRET_KEY`、systemd、nginx 和调度器配置。

## 0. 关键原则

- `.env` 不入 git，权限必须是 `600`。
- `WP_SECRET_KEY` 必填，用于加密 SQLite 中的机器人 `app_secret`。生成方式：

```bash
openssl rand -hex 32
```

- `WP_SECRET_KEY` 必须和 `data/workpulse.db` 一起备份。丢失后 DB 中已加密的 `app_secret_enc` 无法解密。
- 同一个飞书 app 的长连接建议只跑一个 DingWei 实例。迁移时必须先停旧实例，再起新实例；并行灰度请使用不同飞书 app。

## 1. 全新部署 B

适用于新机器、新飞书 app、新 SQLite 库。

```bash
git clone <repo-url> feishu-bot-bridge
cd feishu-bot-bridge
deploy/install.sh
```

`deploy/install.sh` 会执行：

1. 检查 `openssl`、`curl`、`sqlite3`、`tmux`、`go` 或现成 `bin/workpulse`。
2. 构建 `bin/workpulse`。
3. 如果 `.env` 不存在，从 `.env.example` 生成，并自动写入 `WP_SECRET_KEY=$(openssl rand -hex 32)`。
4. 交互填写飞书 `APP_ID` / `APP_SECRET` / 初始管理员口令。
5. `chmod 600 .env`。
6. 安装 `deploy/workpulse.service` 到 systemd，替换 `{{WORKDIR}}` / `{{USER}}`。
7. `systemctl enable --now workpulse`。
8. 执行 `deploy/smoke.sh`。

安装完成后：

```bash
sudo systemctl status workpulse
sudo journalctl -u workpulse -f
```

后台入口默认经 nginx 暴露为：

```text
https://YOUR_DOMAIN/dingwei/admin/login
```

进入后台后继续：

1. 在“机器人管道”确认 UnifiedRobot `app_secret` 已设置。
2. 在“成员”刷新候选人并录入成员。
3. 在“API key”签发 `key_id + secret`，绑定成员飞书账号。
4. 分发 `tools/sessionhelper/run.sh` 给成员接入会话。

## 2. 迁移部署 A

适用于把现有 DingWei 搬到新机器。

必须搬迁：

```text
data/workpulse.db
.env
```

其中 `.env` 里的 `WP_SECRET_KEY` 必须保持原值；不能重新生成。否则旧 DB 中加密的机器人 `app_secret_enc` 解不开。

推荐步骤：

```bash
# 旧机器
sudo systemctl stop workpulse
tar czf workpulse-migrate.tgz .env data/workpulse.db data/schedule-backup 2>/dev/null || true

# 新机器
git clone <repo-url> feishu-bot-bridge
cd feishu-bot-bridge
tar xzf /path/to/workpulse-migrate.tgz
chmod 600 .env
deploy/install.sh
```

注意：

- 先停旧实例再起新实例，避免同一飞书 app 长连接抢事件。
- 如果必须新旧并行，请给新实例使用另一个飞书 app，并在后台录入新的机器人管道。
- 迁移后先跑 `deploy/smoke.sh`，再切 nginx。

## 3. nginx

仓库提供反代片段：

```text
deploy/nginx-dingwei.conf
```

放入你的 HTTPS `server {}` 中，并按实际端口调整 `proxy_pass`。默认假设：

```text
WP_ADDR=127.0.0.1:8791
```

关键点：

- `/dingwei/` 反代 DingWei HTTP 页面和 webhook。
- `/dingwei/ws/` 反代 sessionHelper WebSocket。
- WS 必须包含 `Upgrade` / `Connection`。
- 必须透传 `X-Real-IP` 和 `X-Forwarded-For`。服务端只在请求来自本机可信反代时信任这些头，防伪造。
- `proxy_read_timeout 3600s`，避免长连接被 nginx 短超时断开。

启用示例：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 4. 调度器 claude-deepseek

R8/R9 的团队排期协调和定时佐证依赖系统级 CLI。

> ⚠️ **R14(P1)后本节将改**：`WP_SCHEDULE_TEAM_FILE / PERSONAL_DIR / BACKUP_DIR` 及 `WP_SCHEDULE_EXPORT` 相关废弃 —— 排期改为 **SQLite 版本化唯一真源(`schedule_doc`)+ 多项目(`project` 表)+ 门户 DB 动态渲染**,不再配 md 文件路径/备份目录;`WP_SCHEDULER_CLI`/`NOTIFY_*`/`TRANSCRIPT`/`EVIDENCE_CRON` 迁到项目行或全局默认。下方为 **R14 前(R8 时代)配置**,P1 落地后按 R14 更新。详见 `docs/交办-多项目组排期-R14.md`。

`.env` 中相关项（R14 前）：

```dotenv
WP_SCHEDULER_CLI=claude --print
WP_SCHEDULER_CONFIG_DIR=/home/workpulse/.claude-deepseek
WP_SCHEDULE_TEAM_FILE=/srv/workpulse/docs/AI研究任务排期/AI-研究工作内容清单.md
WP_SCHEDULE_PERSONAL_DIR=/srv/workpulse/docs/AI研究任务排期
WP_SCHEDULE_BACKUP_DIR=/srv/workpulse/data/schedule-backup
WP_SCHEDULE_NOTIFY_CHAT=oc_xxx
WP_SCHEDULE_NOTIFY_BOT=unifiedrobot
WP_SCHEDULE_EVIDENCE_CRON="0 0,6 * * *"
WP_SCHEDULE_EVIDENCE_TZ=UTC
WP_TRANSCRIPT_DIRS=/srv/workpulse/data/transcripts
```

`WP_SCHEDULE_NOTIFY_CHAT`、`WP_SCHEDULE_NOTIFY_BOT`、`WP_TRANSCRIPT_DIRS`、`schedule.evidence_cron`、`schedule.evidence_tz` 也可在后台“运行配置”预置项中覆盖。

新机需要单独安装并配置 `claude-deepseek`，例如把 DeepSeek credentials 放到：

```text
~/.claude-deepseek/credentials
```

敏感 key 只放在 CLI 配置或系统 Secret，不写入仓库。

## 5. 冒烟

安装脚本会自动执行：

```bash
deploy/smoke.sh
```

也可以手工重跑：

```bash
cd /srv/workpulse
deploy/smoke.sh
```

检查项：

- `/healthz` 返回 200。
- `/admin/login` 返回 200。
- 未登录访问 `/admin` 返回 302/303。
- 使用 `.env` 中初始管理员口令登录。
- POST `/webhook/unifiedrobot` 返回 202，消息队列 `queued=0` 且 done 不回退。
- systemd journal 最近日志包含 `scheduler evidence cron configured`，若非 systemd 环境会给 WARN。

## 6. Docker 构建验证

本地仍可构建镜像：

```bash
docker build -t workpulse:rc .
```

Docker 运行时也必须提供 `WP_SECRET_KEY`，并挂载持久化 `/data`：

```bash
docker run --rm -p 8080:8080 \
  -v workpulse-data:/data \
  --env-file .env \
  workpulse:rc
```

## 7. 常见故障

- 后台机器人管道显示 `app_secret` 未设置：检查 `.env` 是否有 `WP_SECRET_KEY`，启动日志是否有 “cannot be stored without WP_SECRET_KEY”。
- 迁移后飞书 WS 不收消息：确认旧实例已停；同一 app 不能多实例长期并行。
- 后台登录失败：`WP_ADMIN_INIT_PASSWORD` 只在 DB 没有管理员时生效；已有管理员请用既有密码或处理 DB。
- sessionHelper 连不上：确认 nginx `/dingwei/ws/` 配置了 Upgrade 头和 3600s 超时。
- 佐证不触发：检查后台运行配置 `schedule.evidence_cron` / `schedule.evidence_tz`，以及 journal 中的 `scheduler evidence cron configured`。
