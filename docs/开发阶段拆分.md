# WorkPulse 开发阶段拆分

> 目标：按可编码、可评审、可测试的闭环推进。每个阶段完成后先提交 `home` 会话做 CR；全部阶段通过后，再交 `openapi` 会话做端到端测试。
>
> 定性：本文是**实施顺序拆分**，不是产品范围降级。规范锁定的 M0-M9 全量能力仍在范围内；阶段里的“先不做/排后”只表示构建顺序靠后，除非用户另行明确降级。

## 协作规则

- 每个阶段只做该阶段范围内的代码、测试和必要文档，不跨阶段扩散。
- 阶段完成标准：代码已实现、单元/集成测试通过、`go test ./...` 通过、工作区无无关产物。
- 通知 `home` 评审时使用 `tmux send-keys -t home "...评审内容..." Enter`，必须带最后的 `Enter`。
- `home` 非 LGTM 时，修复后再次通知 `home`；拿到 LGTM 后再进入下一阶段。
- 所有阶段都通过后，通知 `openapi` 做测试验收。

## 阶段 0：基线整理与已有改动收口

目标：确认当前脚手架和已提交 M1 排期代码状态，把基线收口到可继续开发状态。

范围：
- 核对 `origin/main` 已提交的 M1 排期实现（`c3c3ae7`）与当前工作区状态。
- 补齐 M1 排期现有代码的测试缺口，如 migration/Repository 一致性、pending 过期、匹配不到、冲突检测等。
- 确认 SQLite migration 与 Repository 接口一致。
- 跑通 `go test ./...`、`go vet ./...`、`go build ./...`。

不做：
- 不接飞书真实 webhook。
- 不做 M8 路由平台。
- 不做 AI 佐证。

交付给 home：
- M1 现有实现说明。
- 测试结果。
- 已知遗留项。

## 阶段 1：M0 最小消息入口与本地可测总线

目标：建立不依赖飞书真实环境的消息收发闭环，供业务模块接入。

范围：
- 定义统一 inbound/outbound `Message` 模型和状态机。
- 实现 DB 队列最小版本：入队、取待处理、标记完成/失败、幂等去重。
- 提供 fake Feishu webhook/发送器，支持本地 E2E 测试。
- `/webhook/{channel}` 从占位变为：接收 JSON -> 归一化 -> 入 inbound 队列。
- outbound 发送先用 fake sender 记录发送结果。

测试重点：
- 重复 `feishu_msg_id` 不重复入队。
- 服务重启后未处理消息仍可处理。
- 入队/出队状态转换正确。

交付给 home：
- M0 最小闭环说明。
- DB 队列语义说明。
- 幂等/重启恢复测试结果。

## 阶段 2：M1 排期指令接入消息链路

目标：让用户通过消息完成排期新增、删除、改期、顺延、全量替换，并走 diff + 确认。

范围：
- 将 `schedule.Service` 接入 router/consumer。
- 支持显式指令：`+`、`-`、`改`、`顺延`、`全量`、`确认`、`取消`。
- pending 过期与同一 actor 单 pending 策略明确化。
- 输出纯文本回执，便于飞书聊天展示。
- 写审计日志。

测试重点：
- 新增 -> diff -> 确认 -> 入库 -> 回执。
- 取消后不入库。
- 多 pending 覆盖旧 pending。
- 关键词匹配不到的提示。
- 日期跨年、单日、顺延、全量替换。

交付给 home：
- M1 消息链路 E2E 测试结果。
- pending 状态机说明。

## 阶段 3：M2 进度、M6 风险、M5 基础查询

目标：形成工作管理的日常使用最小闭环。

范围：
- 进度/完成/结果上报。
- 风险/风险解除。
- 查询：我的排期、我的进度、本周团队概览。
- 纯文本输出，暂不做复杂门户。
- 角色权限：member/collaborator/manager 的最小读写边界。

测试重点：
- 成员只能写自己的进度/风险。
- collaborator 不能改排期。
- manager 可查看全局但默认不代改。
- 进度历史 append、展示取最新。

交付给 home：
- 权限矩阵。
- 进度/风险/查询样例输出。

## 阶段 4：M5 最小门户与运行状态页

目标：让管理者可以在 Web 上查看团队状态，同时保持后台能力最小化。

范围：
- 静态/服务端 HTML 门户：团队排期、进度看板、风险列表。
- `/admin` 保留登录与基础状态。
- 最近消息只读列表。
- 不做完整配置 CRUD，不做复杂可视化。

测试重点：
- 未登录不可访问管理页。
- 门户数据与 DB 一致。
- 最近消息不泄露敏感凭证。

交付给 home：
- 页面路径与截图/HTML 检查结果。
- 安全访问说明。

## 阶段 5：M7 提醒与调度

目标：补齐日常运营提醒，验证系统能主动产生 outbound 消息。

范围：
- 每日个人任务提醒。
- deadline 预警。
- 风险周知。
- fake clock 支持可重复测试。

测试重点：
- 到点触发且不重复提醒。
- 无任务/已完成任务不提醒。
- 发送失败进入重试/失败状态。

交付给 home：
- fake clock 调度测试结果。
- 提醒文案。

## 阶段 6：M8 最小字头 WS 路由

目标：实现消息路由平台的最小可用子集，不一次性吃掉全部通配符复杂度。

范围：
- API key 签发/吊销/绑定账号。
- 普通 prefix 字头注册，不做 `*`/`?` 通配符。
- WS 长连接在线转发。
- WS 断连：不补发，回原发送方“该消息投递失败 + 原消息内容”。
- 账号范围校验。

测试重点：
- API key hash 存储，明文只返回一次。
- 未绑定账号不转发。
- WS 在线转发成功。
- WS 断连不入待转发队列、不重放。

交付给 home：
- M8 最小语义说明。
- 安全/断连测试结果。

## 阶段 6b：M8 字头通配符与覆盖检测

目标：补齐规范锁定的字头通配符与覆盖检测能力，作为封版前硬门。

范围：
- 支持 `?` 可选单字符、`*` 任意多字符、`;`/`；` 多字头 OR。
- 支持大小写敏感配置、strip prefix 配置。
- 注册/修改路由时做同作用域内唯一性与覆盖/重叠检测。
- 明确冲突策略：默认拒绝重叠；允许优先级消歧时必须显式配置并在后台高亮。

测试重点：
- `openAPI???` 匹配 0-3 个后缀字符。
- `openAPI???` 与 `openAPI1` 判定重叠。
- `open*` 与 `openAPI` 判定重叠。
- `*` 级宽通配触发高风险警示。
- 不同账号范围/渠道作用域不误判冲突。

交付给 home：
- pattern 相交检测说明。
- 覆盖检测测试矩阵。

## 阶段 7：M4 AI 佐证最小闭环

目标：先用 fake transcript adapter 跑通抽取、映射、对账，再考虑真实 AI CLI transcript。

范围：
- evidence adapter 接口。
- fake adapter 测试数据。
- LLM provider 抽象，测试用 fake provider。
- 写 `ai_evidence` 和 `reconciliation`。
- 生成五态对账结果。
- opt-out 生效。

测试重点：
- opt-out 后不抽取、不写 evidence。
- 原文不入库，只存摘要和 hash。
- 自报完成 + 无证据 -> suspected_lag。
- 无自报 + 有证据 -> ahead。

交付给 home：
- AI 佐证隐私边界说明。
- fake adapter 对账样例。

## 阶段 7b：M9 完整配置 CRUD 与管理者代改确认

目标：补齐规范锁定的管理后台完整配置能力，以及管理者代改他人排期的确认/通知/申诉闭环。

范围：
- M9 完整配置 CRUD：成员、机器人管道、路由字头、API key、账号绑定、LLM、留存策略。
- 所有配置写 SQLite，热更新生效，非法配置先校验再拒绝写入。
- 管理者代改他人排期：默认需二次确认，通知被影响成员，成员可确认/申诉，审计留痕。
- 最近 100 条消息筛选与状态展示。

测试重点：
- 管理后台写配置后无需重启生效。
- 非法配置不落库。
- API key 明文只显示一次，库内只存 hash。
- 管理者代改不会静默改动成员排期。
- 被影响成员能收到通知并看到变更来源。

交付给 home：
- M9 CRUD 范围清单。
- 管理者代改状态机与测试结果。

## 阶段 8：封版联调与交给 openapi 测试

目标：整体验收并交给 `openapi` 会话测试。

范围：
- 全量 `go test ./...`、`go vet ./...`、`go build ./...`。
- Docker build。
- 本地 fake E2E 测试脚本。
- 整理部署/测试说明。
- 汇总 home 各阶段 LGTM 结论。
- 确认阶段 6b 与 7b 已完成或已获得用户明确降级，不允许默默遗漏规范锁定项。

交付给 openapi：
- 测试入口。
- 测试账号/样例消息。
- 已知限制。
- 回归清单。

## 排后项

- Redis Stream 队列后端：可选扩展，DB 队列先满足 MVP 规模。
- 飞书交互卡片：体验增强项，纯文本先闭环。
- 真实 AI CLI transcript 适配器：先用 fake adapter 验证 M4 链路，真实适配器靠后接入。

## 封版前硬门

- 字头通配符 `*`/`?` 与覆盖检测完整实现（阶段 6b）。
- M9 完整配置 CRUD（阶段 7b）。
- 管理者代改他人排期的确认/通知/申诉流程（阶段 7b）。
- M3 变更联动：排期/进度变更后协调更新全局日程并同步全局日程文件。
- M3/M7 影响通知：关键节点顺延、deadline 改变、依赖打断等明显影响必须通知所有当事人和管理者。
- `/portal` 不得裸露敏感团队信息：封版前必须加可配置鉴权，或部署文档明确仅内网/反向代理鉴权访问。
- 消息入库前脱敏：token/password/secret/authorization/api_key/Bearer/sk-* 等敏感值不得以明文进入主库。
