# sessionHelper 忙时消息缓存规格 v0.1

> 目标：CLI 忙时收到的消息不丢——缓存 + 回执"稍后处理" + 空闲后逐条灌入。

## 一、需求
1. sessionHelper 收到可注入消息,**CLI 空闲**→立即灌入(现行为);
2. **CLI 忙**(正处理任务)→ ①**缓存进队列** ②立即回发件方:`消息已收到，但此时忙，稍后处理你的请求`;
3. CLI **转空闲后** → **FIFO 逐条灌入**缓存队列(保序、不丢);
4. 适用:路由来的协作消息(`#会话名`/send.py)+ 网页输入;**不适用** skill 指南/在线清单(它们特殊处理、无需缓存)。

## 二、忙/闲判定
- 复用 cli.py 现有 `is_idle()`(距上次输出 ≥ `cli_settle_seconds` 且已 ready)= 闲;否则 = 忙;
- PTY 模式:CLI 在产出/未回到就绪 prompt = 忙;
- 判定要稳(避免抖动):建议"连续 idle ≥ settle" 才算真闲再开灌。

## 三、机制
```
收到可注入消息 env:
  if adapter.is_idle():
      inject(env)                        # 立即灌
  else:
      pending.append(env)                # 入缓存队列(FIFO)
      if not already_replied_busy(from): # 每个发件方忙回执只发一次(或合并),防刷屏
          reply(from, "消息已收到，但此时忙，稍后处理你的请求")

后台 drain 循环(或 idle 边沿触发):
  当 adapter 由忙转闲:
      while pending and adapter.is_idle():
          inject(pending.popleft())      # 逐条灌
          等该条被 CLI 接收/开始处理(避免一次糊一堆) 再灌下一条
```

## 四、细节与边界
- **队列上限**:设上限(如 100),超限丢最旧并日志(防忙时堆爆);
- **回执节流**:同一发件方忙期内只回一次"忙"(或 N 秒合并),别每条都回;
- **保序**:严格 FIFO;
- **灌入节奏**:每灌一条,等 CLI 真正接收/开始(短暂 settle)再灌下一条,避免连续注入被 CLI 吞键;
- **回执路由**:忙回执发回 `from`(会话间走 send.py/agent_route 那套地址;飞书来源回飞书);
- **与总控配合**:总控侧已有"入队即 ack",本机忙缓存是**会话本地的第二层缓冲**,两者互补(中央保交付 + 本机忙不丢)。

## 五、跨平台(Linux + macOS)
- 本功能是**纯 Python 逻辑**(队列 + `is_idle()` + 注入),不涉及 OS 特定调用,**Linux/macOS 天然通用**;
- 验收应在 Linux 与 macOS 各运行一遍，确认 PTY 忙/闲判定在两平台都稳定。

## 六、验收要点
1. CLI 忙时发消息 → 发件方收到"忙"回执 + 消息进缓存(不丢);
2. CLI 忙完 → 缓存消息**按序逐条**被灌入 CLI 并处理;
3. 忙期多条来自同一方 → 只回一次"忙";
4. 队列超上限 → 丢最旧 + 日志,不崩;
5. 空闲时发消息 → 仍立即灌(无回归);
6. skill/清单消息 → 不进此缓存(照旧特殊处理)。
